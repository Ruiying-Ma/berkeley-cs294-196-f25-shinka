# GLOBAL INSIGHTS SCRATCHPAD

The following are global insights about optimization approaches and their effectiveness:

Based on the analysis of the individual program summaries and the code of the current best program, here are the actionable optimization insights.

## Successful Algorithmic Patterns
- **Static 1% Admission Window**: The top-performing programs (Generation 62, 64, 65, 66, 67, 69) converged on a fixed 1% admission window. Evaluations explicitly show that increasing this window to 5% (Generation 63, 68) consistently caused the score to drop from 0.27 to 0.26. This indicates that for this dataset, maximizing the Main cache capacity (specifically the Protected segment) is more critical than buffering larger scans.
- **Two-Hit Probation Promotion**: The current best program implements a strict SLRU promotion logic where items in the Probation segment are **not** promoted to Protected on the first hit. Instead, they are tracked in a `probation_hits` structure and only promoted on a subsequent access. This acts as a secondary filter (after the Doorkeeper) to strictly reserve the Protected segment for items with proven high frequency.
- **Main-Biased Tie-Breaking**: The eviction duel in the best code (`if freq_window > freq_main`) determines that if frequencies are equal, the Window item is evicted. This standard incumbency biasretaining the Main item during tiesprovides sufficient stability to maintain high hit rates on static traces (like Trace 28) without the need for the overly aggressive `+1` margin that hurt Generation 63.

## Ineffective Approaches
- **Enlarged Admission Windows (5%)**: Programs that attempted to increase the window size to 5% (Generation 63, 68) failed to outperform the 1% baseline. The feedback suggests that the larger window diluted the cache's ability to retain long-term hot items, leading to a net loss in performance despite theoretically better scan handling.
- **Overly Conservative Aging**: Generation 61 employed a "conservative aging" strategy which led to cache pollution by obsolete high-frequency items. While the best program uses periodic aging (every 5x capacity), making it too slow prevents the cache from clearing out old heavy hitters when the workload shifts, hurting performance on dynamic traces.
- **Runtime-Unsafe State Management**: Generation 60 failed completely because it attempted to access `capacity` in the `update_after_evict` function where it wasn't in scope. This highlights that all algorithmic parameters must be encapsulated within the persistent state object to ensure they are accessible across all lifecycle hooks.

## Implementation Insights
- **Graceful Protected Demotion**: The best program handles overflow in the Protected segment by **demoting** the victim back to the Probation segment rather than evicting it entirely. This keeps the item in the cache and gives it a "second chance" to prove its utility in the Probation queue, preventing the immediate loss of valuable but slightly colder items.
- **Clean Doorkeeper Transitions**: The implementation uses a standard Python `set` for the Doorkeeper. Crucially, when a key moves from the Doorkeeper to the Frequency map, the code explicitly performs `self.doorkeeper.remove(key)`. This ensures the Doorkeeper strictly filters only the very first access and keeps the memory footprint efficient.
- **Forced Window Availability**: The `evict` function in the best code explicitly checks `if len(state.window) >= state.window_size`. If the window is full, it forces a duel or eviction. This guarantees that the window never exceeds its tiny quota, preventing it from eating into the Main cache's capacity which is vital for the SLRU strategy.

## Performance Analysis
- **The 0.27 Score Ceiling**: Multiple distinct implementations (Gen 62, 64-67, 69) have hit a hard ceiling at a score of 0.27. These programs are characterized by excellent performance on static, repetitive traces (e.g., Trace 28 hit rate ~0.89) but near-zero performance on loop/scan traces (Trace 14 = 0.00).
- **Scan vs. Locality Trade-off**: The evaluations reveal a strict trade-off. Attempts to capture the scanning patterns of Trace 14 (via larger windows or looser admission) inevitably reduce the hit rate on Trace 28. Since Trace 28 offers a higher potential hit rate ceiling, the optimizer favors "specializing" in locality over scans.
- **Triple Filtering Efficacy**: The success of the current best architecture relies on a pipeline of three filters: 1) Doorkeeper (filters first access), 2) Admission Duel (filters entry to Main), and 3) Probation Promotion (filters entry to Protected). This aggressive multi-stage filtering is what allows a 99% Main cache to maintain high purity and achieve the 0.27 score.

# META RECOMMENDATIONS

The following are actionable recommendations for the next program generations:

Based on the global insights and the analysis of the current best program (Generation 69), here are 5 actionable recommendations for future mutations:

1.  **Grant "Second Chance" Status on Demotion**
    Modify the `update_after_hit` logic to set `state.probation_hits[victim] = True` immediately when an item is demoted from the `protected` segment to `probation`. This acknowledges the item's previous high status and allows it to return to the protected segment with a single subsequent hit, reducing churn for the working set boundary compared to treating it as a raw newcomer.

2.  **Decouple Doorkeeper Reset from Frequency Aging**
    Change the `doorkeeper` clearing mechanism to trigger based on a size threshold (e.g., when `len(state.doorkeeper) > 2 * capacity`) rather than clearing it inside the periodic `age()` function. This prevents the filter from suffering "amnesia" on recently seen scan items just because the frequency aging timer fired, thereby improving scan resistance.

3.  **Lower Frequency Saturation Cap**
    Reduce `self.max_freq` from 60 to **15**. The current cap of 60 retains history for too long, causing the cache to hold onto "ghost" heavy hitters that are no longer relevant; a lower cap ensures the frequency sketch adapts faster to phase changes while still effectively distinguishing between one-hit wonders and established items.

4.  **Implement Segment-Aware Eviction Duel**
    Refine the `evict` function to use different tie-breaking logic based on the victim's segment. Use `freq_window >= freq_main` (ties evict Main/Probation) if the victim is in the **Probation** segment, but strictly `freq_window > freq_main` (ties keep Main/Protected) if the victim is in the **Protected** segment. This encourages faster turnover in the probation buffer while strictly stabilizing the core protected segment.

5.  **Frequency-Accelerated Promotion**
    Update `update_after_hit` to allow immediate promotion from `probation` to `protected` if `state.get_freq(key)` exceeds a "hot" threshold (e.g., 5), even if `probation_hits` is not satisfied. This utilizes the persistent frequency sketch to fast-track historically heavy hitters that have re-entered the cache, bypassing the standard two-hit probation requirement.
