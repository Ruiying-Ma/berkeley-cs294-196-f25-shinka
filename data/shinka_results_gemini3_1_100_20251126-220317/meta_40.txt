# GLOBAL INSIGHTS SCRATCHPAD

The following are global insights about optimization approaches and their effectiveness:

# Successful Algorithmic Patterns
- **Preemptive Main Eviction for Window Availability:** The Current Best Program (Score 0.26) and comparable top performers (Gen 35, 39) implement a critical logic: `if window_size < w_cap: evict from Main`. By preferentially evicting from the Main cache (Probation/Protected) when the admission window is not full, the algorithm ensures the window acts as a high-throughput buffer that is always open to new arrivals. This prevents the admission window from becoming a bottleneck where new items are rejected before they can be sampled.
- **Segmented LRU (SLRU) with Frequency Dueling:** The most robust programs (Gen 35, 38, 39) consistently use a Segmented LRU for the main cache, typically splitting it into Probation (~20%) and Protected (~80%). The admission to Probation is guarded by a frequency duel (`freq_window > freq_main`), and promotion to Protected requires a subsequent hit. This structure maximizes the retention of items that have proven their value (frequency + recency) while filtering one-off scans.
- **Long-Horizon Periodic Aging:** Unlike programs that aged frequencies too aggressively (Gen 36, every `capacity` accesses), the successful variants (Gen 38, Current Best) employ a slower aging schedule, typically halving frequencies every `10 * capacity` accesses. This retains historical frequency data long enough to recognize looping patterns that slightly exceed cache size (Trace 28 hit rate ~0.89) without allowing ancient history to pollute the cache indefinitely.

# Ineffective Approaches
- **Rapid Frequency Aging:** Gen 36 attempted to age frequencies every time the access count equaled the cache capacity. This proved detrimental (Score 0.24), causing complete failure (0.00 hit rate) on traces where the loop size was slightly larger than the cache. The rapid decay erased the "ghost" frequency data before the items could return, neutralizing the LFU benefits.
- **Strict and Low Frequency Caps:** Gen 36 also applied a strict cap of 15 to frequency counts. This low ceiling prevented the algorithm from distinguishing between moderately popular items and "super-hot" items in skewed workloads, leading to inefficient eviction decisions and higher variance compared to the uncapped or high-cap implementations.
- **Hybrid S3-FIFO Complexity:** Approaches attempting to blend S3-FIFO with TinyLFU mechanics (Gen 31, Score 0.22) or add granular frequency counters to S3-FIFO (Gen 37, Score 0.24) failed to outperform the cleaner W-TinyLFU structure. The overhead of managing multiple FIFO queues with lazy promotion or 3-bit counters did not provide better discrimination than the direct frequency duel of the best program.

# Implementation Insights
- **Direct Frequency Dueling Logic:** The Current Best Program's effectiveness hinges on the explicit comparison: `if freq_w > freq_m: evict_main else evict_window`. This direct duel, executed only when the window is full, serves as a strict filter. Importantly, it is coupled with the `update_after_insert` logic that moves items to Probation immediately upon window overflow, ensuring the duel only happens when strictly necessary.
- **Precise Python Structures:** The top-performing code leverages `collections.deque` for the admission window and `collections.OrderedDict` for the SLRU segments. # META RECOMMENDATIONS

The following are actionable recommendations for the next program generations:

Based on the analysis of the current best program (W-TinyLFU variant) and the global insights, here are 5 actionable recommendations for the next program mutation:

1.  **Implement a "Doorkeeper" Mechanism**
    To address the poor performance on scan traces (Trace 14, 0.00 hit rate), introduce a `set` to track newly observed keys. When an item is accessed, if it is not in the `set`, add it but do not increment the main `freq` dictionary (effectively keeping its frequency at 0 for the duel). Only increment `freq` if the key is already in the `set`. This ensures one-off items lose the eviction duel (`freq 0` vs `freq > 0`) against the working set, preventing cache pollution.

2.  **Condition Preemptive Eviction on Frequency**
    The current logic (`if window_size < w_cap: evict from Main`) unconditionally prioritizes keeping the window open, which can flush valuable Main items during a scan. Modify this to check the Main victim's status. If the candidate for eviction from Main has a high frequency (or is in the `protected` segment), **skip** the preemptive eviction and allow the window to fill. This forces a proper frequency duel when the window overflows, ensuring hot Main items are not sacrificed for cold Window arrivals.

3.  **Accelerate Frequency Aging to 5x Capacity**
    The current aging interval (`10 * capacity`) is successful but likely too slow for rapid phase changes (Traces 13, 17), while the failed `1 * capacity` was too aggressive. Change the aging trigger to `access_count > capacity * 5`. This intermediate value will clear obsolete historical frequency data faster than the current implementation without destroying the retention of active loops.

4.  **Raise Admission Threshold for "Protected" Segment**
    Currently, a single hit in `probation` promotes an item to `protected` (80% of cache). Increase this requirement: promote to `protected` only if an item has received **2 or more hits** while in `probation`. This stricter filter ensures that the large protected sanctuary is reserved for items with proven stability, keeping the `probation` segment more fluid for testing new content.

5.  **Implement a Soft Frequency Cap**
    To prevent "historical inertia" where old heavy hitters remain immune to eviction long after they become cold, introduce a cap on the frequency counters (e.g., `min(freq, 60)`). A cap of ~60 is high enough to distinguish popular items but low enough to allow the `5x` aging mechanism to degrade their counts effectively when they stop being accessed.
