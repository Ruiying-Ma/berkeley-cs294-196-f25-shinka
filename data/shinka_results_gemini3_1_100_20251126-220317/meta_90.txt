# GLOBAL INSIGHTS SCRATCHPAD

The following are global insights about optimization approaches and their effectiveness:

Based on the analysis of the individual program summaries (Generations 80-89) and the Current Best Program, here are the actionable optimization insights.

## Successful Algorithmic Patterns
- **Implicit Incumbent Favoritism**: The **Current Best Program** (Score 0.27) uses a strict inequality (`freq_window > freq_main`) during eviction duels. This implicitly breaks ties in favor of the incumbent Main cache item. Comparative analysis shows that **Generation 82** (Score 0.26) and **Generation 84** (Score 0.26) tried to improve this with explicit biases ("+5 demotion bias" and "Ghost Shield"), but these modifications slightly reduced the score compared to the cleaner, implicit approach.
- **Secondary Probation Filtering**: The **Current Best Program** enforces a "2-hit" rule within the Probation segment before promoting an item to the Protected segment. **Generation 88** (Score 0.26) attempted a "Fast Promotion" mechanism to bypass this for high-frequency items but scored lower. This confirms that a strict secondary filterpreventing noise from polluting the Protected segmentis superior to aggressive promotion.
- **Demotion to Probation**: A critical pattern in the best implementation is handling Protected segment overflow by demoting items to the Probation segment rather than evicting them. This gives "warm" items a second chance to compete against new Window candidates. **Generation 80** (Score 0.23) applied strict frequency-based admission without this graceful demotion flow (or with too much bias), leading to worse performance.

## Ineffective Approaches
- **Explicit Arithmetic Biasing**: Several programs attempted to artificially boost incumbent frequencies. **Generation 83** added a `+1` bias, and **Generation 80** used strong protection biases, resulting in scores of 0.24 and 0.23 respectively. Evaluation feedback indicates these explicit biases make the cache too rigid, causing it to fail on dynamic workloads where the **Current Best Program's** natural tie-breaking succeeds.
- **Window Size Expansion**: **Generation 85** (Score 0.24) increased the Window size to 5% to better handle scanning workloads. While this theoretically helps scans, the score dropped significantly compared to the **Current Best Program** (1% Window). The loss of Main cache capacity (reduced from 99% to 95%) outweighs the benefits on scan traces for this specific dataset.
- **Aggressive Admission Filtering**: **Generation 81** (Score 0.23) combined a 1% window with aggressive admission filtering. The feedback noted that this combination was too strict, evicting new items before they could prove their worth. The **Current Best Program** balances the 1% window with a standard Doorkeeper and fair aging (5x capacity) rather than aggressive rejection.

## Implementation Insights
- **Separate Probation State Tracking**: The **Current Best Program** utilizes a dedicated auxiliary dictionary, `probation_hits`, to track hit counts for items within the Probation segment. This implementation detail allows for stateful promotion logic (requiring 2 hits) without polluting the main object structure or the primary cache storage, facilitating cleaner state transitions compared to **Generation 82**.
- **Graceful Demotion Reset**: In the **Current Best Program**, when a Protected item is demoted, it is moved to the Probation segment (`state.probation[victim] = True`) but notably *without* an entry in `probation_hits`. This effectively resets its promotion eligibility, requiring it to prove itself again, which serves as a robust mechanism to filter out items that were once hot but have cooled down.
- **Strict Window Capacity Enforcement**: The `evict` function in the **Current Best Program** strictly checks `if len(state.window) >= state.window_size` to force an eviction decision. This implementation guarantees the Window never silently expands beyond 1%, preserving the exact 99% capacity for the SLRU structure, which is critical for the high hit rates (0.89) on Trace 28.

# META RECOMMENDATIONS

The following are actionable recommendations for the next program generations:

Based on the analysis of the global insights and the current best program (Generation 69), here are 5 actionable recommendations for future mutations.

1.  **Grant "Second Chance" Status on Demotion**
    Modify the `update_after_hit` logic to set `state.probation_hits[victim] = True` immediately when an item is demoted from `protected` to `probation` due to overflow. Currently, demoted items are treated as raw newcomers requiring two hits to return; granting them "second chance" status allows these proven items to return to the protected segment with a single subsequent hit, reducing churn at the working set boundary.

2.  **Lower Frequency Saturation Cap**
    Reduce `self.max_freq` from 60 to **15** or **20**. The current cap creates too much inertia, causing "ghost" heavy hitters to persist long after they have stopped being relevant. A lower saturation point, combined with the existing 5x aging interval, will significantly improve the cache's ability to adapt to phase changes (dynamic workloads) while still maintaining the "Implicit Incumbent Favoritism" pattern.

3.  **Implement Segment-Dependent Eviction Logic**
    Refine the `evict` function to use different tie-breaking logic based on the victim's segment. Use `freq_window >= freq_main` (ties evict Main) if the victim is in the **Probation** segment, but maintain strictly `freq_window > freq_main` (ties keep Main) if the victim is in the **Protected** segment. This encourages faster turnover in the probation buffer for new content while strictly stabilizing the core protected segment against replacement.

4.  **Sketch-Accelerated Promotion**
    Update `update_after_hit` to allow immediate promotion from `probation` to `protected` if `state.get_freq(key)` exceeds a significant threshold (e.g., 5 or 10), even if the `probation_hits` condition is not met. This utilizes the persistent frequency sketch to fast-track historically heavy hitters that are re-entering the cache, allowing them to bypass the standard "two-hit" probationary requirement intended for unknown items.

5.  **Size-Based Doorkeeper Reset**
    Decouple the doorkeeper clearing mechanism from the frequency aging timer. Add a check in `record_access`: if `len(state.doorkeeper) > 2 * capacity`, clear the set immediately. This prevents the admission filter from becoming "clogged" with one-hit wonders during long periods between aging intervals, ensuring it remains an effective filter against scanning workloads.
